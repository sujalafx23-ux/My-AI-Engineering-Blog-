What is Artificial Intelligence?

Wikipedia, IBM

Artificial Intelligence is software that can perform tasks that normally require human-level intelligence — reading, writing, reasoning, recognising images, translating languages. The key word is "normally." AI doesn't think the way humans do — it learns patterns from enormous amounts of data.

💡 Analogy

Imagine showing a child 10,000 pictures of cats labelled "cat," and 10,000 pictures of dogs labelled "dog." The child learns what features cats and dogs have. AI training is the same — billions of examples, millions of repetitions, until it recognises patterns reliably.

Types of AI

🏹
Narrow AI
Does one specific thing well. Spam filters, chess engines, face unlock. Almost all AI today is this.
✨
Generative AI
Creates new content — text, images, code, audio. ChatGPT, Claude, DALL-E, Sora, Midjourney are all generative AI.
🌌
AGI
AI that can do anything a human can. Does not exist yet. Still theoretical research goal.

How AI Learns: The Training Process

1
Collect Massive Data
Billions of sentences from the internet, books, Wikipedia, code repositories. For GPT-3 this was ~300 billion tokens from Common Crawl, WebText, Wikipedia and books.
2
Pre-Training (Self-Supervised)
The model tries to predict the next word in a sentence — billions of times. It checks if it was right, adjusts internal numbers (weights), and repeats on thousands of GPUs for weeks or months.
3
RLHF — Fine-Tuning with Humans
Human raters compare two model responses and say which is better. The model learns to prefer more helpful, accurate, and safe outputs. Used by both OpenAI and Anthropic.
4
Deploy via API
The trained model is hosted on servers. You interact with it through a website (ChatGPT, Claude.ai) or programmatically through an API.

Large Language Models (LLMs)

Wikipedia, DeepLearning.AI

LLM = Large Language Model. It's the type of AI behind ChatGPT, Claude, and Gemini. "Large" refers to the number of parameters — internal numerical values that the model learns during training. More parameters generally = more capability (but also more cost and compute).

ℹ️ KEY CONCEPT

LLMs are fundamentally next-token predictors. They don't understand language the way humans do — they learn extremely rich statistical patterns over trillions of tokens. The outputs look intelligent because the patterns are so deep.

Parameter Counts — The "Large" in LLM

Model Params By Year Notes
GPT-2 1.5B OpenAI 2019 First publicly released large model
GPT-3 175B OpenAI 2020 Launched the "foundation model" era
GPT-4 ~1T (est.) OpenAI 2023 Multimodal; architecture undisclosed
Claude 3.5 Sonnet Undisclosed Anthropic 2024 200K context window
Llama 3.1 8B–405B Meta 2024 Open-weight, runs locally
Gemini 1.5 Pro Undisclosed Google 2024 1M token context window

Two Main Types of LLMs

Decoder-Only (GPT-style)

  • Generates text left-to-right
  • Used for: chat, code, creative writing
  • Examples: GPT-4, Claude, Llama 3
  • Trained on "next token prediction"

Encoder-Only (BERT-style)

  • Reads and classifies text
  • Used for: sentiment, search, classification
  • Examples: BERT, RoBERTa, DistilBERT
  • Trained on "masked token prediction"

🔬 FROM RESEARCH

According to Wikipedia's LLM article, as of 2024, the largest and most capable models are all based on the decoder-only Transformer architecture. LLMs like GPT-4 and Claude demonstrate "emergent behaviours" — capabilities that appeared unexpectedly at scale that weren't explicitly trained for, like multi-step reasoning and code generation.

The Transformer Architecture

Attention Is All You Need, Sebastian Raschka, IBM

Every modern LLM is built on the Transformer architecture, introduced by Google in the 2017 paper "Attention Is All You Need." Before Transformers, language models used Recurrent Neural Networks (RNNs) which processed text one word at a time — slow, and bad at remembering things said far back in a sentence. Transformers process the entire sequence at once using a mechanism called attention.

📄 "Attention Is All You Need" (2017) 📚 transformer-explainer.poloclub.github.io 📚 sebastianraschka.com

The Core Idea: Attention

The attention mechanism lets every word "look at" every other word in the sentence and decide how much weight to give it. Consider:

How Attention Resolves Ambiguity

"The bat flew over the swing" → bat = animal (high attention to "swing", "flew")

"He hit the ball with the bat" → bat = sports equipment (high attention to "hit", "ball")

📚 FROM DATACAMP

Attention mechanisms assign a score to each word based on relevance. Words like "swing" and "flew" get high scores when resolving "bat" as an animal, vs "hit" and "ball" for the sports equipment meaning. This lets the model understand context across the whole sentence simultaneously.

Self-Attention: Q, K, V Matrices

Each token is multiplied by three learned weight matrices to produce Query (Q), Key (K) and Value (V) vectors. The dot product of Q and K gives an attention score — how much this token should "attend to" another token. These scores are scaled, masked (to prevent peeking at future tokens), run through Softmax to become probabilities, then multiplied by V.

Pseudocode · Self-Attention 
# Each word → 3 vectors via learned weight matrices
Q = token_embedding @ W_Q   # "What am I looking for?"
K = token_embedding @ W_K   # "What do I contain?"
V = token_embedding @ W_V   # "What do I contribute?"

# Attention score: how much should token A attend to token B?
scores = (Q @ K.T) / sqrt(d_k)   # Scale to prevent vanishing gradients
scores = masked_fill(scores, future_mask, -inf)  # Can't peek ahead
weights = softmax(scores)         # Probabilities (sum to 1)

# Weighted sum of values = context-aware representation
output = weights @ V

Multi-Head Attention

Instead of one attention calculation, Transformers run multiple attention heads in parallel (GPT-2 small has 12, larger models have 96+). Each head learns to attend to different aspects — one might focus on grammar, another on named entities, another on long-range topic dependencies. Their outputs are concatenated and projected.

Full Transformer Block (Decoder)

Language Model Head (Output) Projects to vocab size → probabilities over 50k+ tokens
Feed-Forward Network (×N layers) Refines each token's representation independently
Multi-Head Masked Self-Attention (×N layers) Tokens attend to all previous tokens — the "magic"
Positional Encoding Adds order info — "this is word 5 of 20"
Token Embedding Layer Words → dense vectors (numbers)
Input: Raw Text "Hello world" → tokenised → [15496, 995]

✅ KEY INSIGHT (from Transformer Explainer, Georgia Tech)

The transformer processes text through three stages: (1) Tokenisation & Embedding, (2) A stack of Transformer blocks (each containing attention + feed-forward layers), (3) A Language Model Head that converts the final representation into a probability over all possible next tokens.

Tokens & Context Windows

OpenAI Tokenizer, IBM, Wikipedia

Tokens are the basic unit LLMs work with — not characters, not words, but chunks. They're determined by a tokenizer (like BPE — Byte Pair Encoding) trained to find the most efficient way to chop up text. One token ≈ 4 characters ≈ ¾ of a word in English.

Same Sentence — Different Tokens

Input: "Transformers are changing everything"

Transform ers are changing every thing

6 tokens · Note "Transformers" = 2 tokens, "everything" = 2 tokens

⚠️ WHY TOKENS MATTER FOR YOU

API pricing is per token (both input and output). Temperature and max_tokens you set in your API call directly control the length of output. Longer prompts = higher cost. Always be aware of how many tokens your prompt uses before deploying at scale.

Context Window — The AI's "Working Memory"

The context window is the maximum number of tokens the model can process in one call — your messages + the AI's replies + any system instructions, all counted together. Once the context is full, older messages are dropped.

Model Context Window Equivalent Use Case
GPT-3.5 Turbo 16K tokens ~12,000 words Short chats, summaries
GPT-4o 128K tokens ~100K words / short novel Long docs, codebases
Claude 3.5 Sonnet 200K tokens ~150K words Entire books, large codebases
Gemini 1.5 Pro 1M tokens ~750K words Hours of video, entire repos

ℹ️ NO PERSISTENT MEMORY

LLMs have no memory between conversations. Each new chat session starts completely fresh. The "memory" features you see in apps (like ChatGPT's memory) are built on top of the API — they inject past conversation summaries into the system prompt. This is important to understand when building apps.

Prompt Engineering

OpenAI Docs, PromptingGuide.ai, Lakera, Anthropic

A prompt is the input you give the AI. Prompt engineering is the skill of designing prompts that reliably produce the outputs you want. According to OpenAI's prompting guide: "AI engineering is inherently an empirical discipline — LLMs are nondeterministic. Build evals and iterate often."

Anatomy of a Complete Prompt

Prompt Structure 
// ① SYSTEM PROMPT — sets AI's role, rules, and tone
"You are a senior Python engineer. Write clean, documented code.
Always add type hints. Raise exceptions with descriptive messages."

// ② FEW-SHOT EXAMPLES — show the pattern you want
"Example:
Input: 'parse CSV'
Output: def parse_csv(path: str) -> list[dict]: ..."

// ③ CONTEXT — give the AI what it needs
"The CSV has columns: name, age, email, signup_date"

// ④ TASK — the actual instruction
"Write a function that reads the CSV and returns
only users who signed up in the last 30 days."

// ⑤ OUTPUT FORMAT — tell it exactly what to produce
"Return only the Python function. No explanation."

Technique 1: Chain-of-Thought (CoT)

Adding "Think step by step" forces the model to reason before answering. Introduced in Wei et al. (2022), CoT dramatically improved accuracy on math and reasoning tasks. Research showed PaLM's score on math word problems rose from 18% to 57% using few-shot CoT prompts.

❌ Without CoT

  • Q: "I had 10 apples, gave 2 to neighbor, 2 to repairman, bought 5 more, ate 1. How many?"
  • → Wrong answer: "12"

✅ With "Think step by step"

  • Start: 10. Gave 2+2=4 away → 6. Bought 5 → 11. Ate 1 → 10.
  • → Correct answer: "10"

📚 FROM PROMPTINGGUIDE.AI

Zero-shot CoT (Kojima et al. 2022): Simply adding "Let's think step by step" to a prompt, without any examples, significantly improves performance on reasoning tasks. Self-Consistency further improves this by sampling multiple reasoning paths and taking a majority vote — raising accuracy on math word problems from 57% to 74%.

Technique 2: Few-Shot Learning

Provide 2–5 examples of the exact input→output pattern you want. The model picks up the pattern and applies it to new inputs. From OpenAI's documentation: "Few-shot prompting lets you steer an LLM toward a new task by including examples, rather than fine-tuning the model."

Python · Few-Shot in API Call 
# Few-shot: classify customer reviews
messages = [
  {"role": "system", "content": "Label reviews as Positive, Negative, or Neutral."},
  {"role": "user",   "content": "The battery life is amazing!"},
  {"role": "assistant", "content": "Positive"},
  {"role": "user",   "content": "It stopped working after 2 days."},
  {"role": "assistant", "content": "Negative"},
  {"role": "user",   "content": "The packaging was okay."},
  # Model now understands the pattern and labels correctly
]

Technique 3: Role + Format Control

🎭
Role Assignment
"You are a cybersecurity expert…" shapes the model's vocabulary, assumptions, and tone for the entire conversation.
📐
Output Format
"Respond only in JSON with keys: name, score, reason." Forces structured output the code can parse reliably.
🛡️
Prompt Scaffolding
Wrap user input in guard templates: "You MUST only discuss [topic]. If asked anything else, politely decline."

✅ LAKERA'S GUIDE — PRO TIPS

Treat complex prompts like UX design. Group related instructions. Use section headers, examples, and whitespace. If a human would struggle to follow it, the model probably will too. Combining prompt types (few-shot + role + CoT + format) is how you go from "it kind of works" to "production-ready."

APIs — Talking to AI Programmatically

OpenAI Docs, Anthropic Docs

An API (Application Programming Interface) lets your code talk directly to the AI model. Instead of typing in a chat window, you send a structured HTTP request with your message, and get a structured response back — all in code.

💡 Analogy

A restaurant kitchen (the AI model) makes food. You don't walk into the kitchen — a waiter (the API) takes your order in a standard format (the HTTP request), goes to the kitchen, and brings back your food (the JSON response). The API is the standard interface between you and the model.

Real Code — OpenAI API

Python · OpenAI 
import os
from openai import OpenAI

# Never hardcode keys! Use environment variables
client = OpenAI(api_key=os.getenv("OPENAI_API_KEY"))

response = client.chat.completions.create(
    model="gpt-4o",          # Which model to use
    messages=[
        {"role": "system",    # Sets AI's behaviour
         "content": "You are a helpful coding assistant."},
        {"role": "user",      # User's message
         "content": "Explain what a decorator is in Python."}
    ],
    temperature=0.7,          # 0 = deterministic, 1 = creative
    max_tokens=800            # Max length of response
)

# Extract the text reply
answer = response.choices[0].message.content
print(answer)

# Check cost: response.usage.total_tokens

⚠️ API KEY SAFETY

Never hardcode API keys in your code. Use environment variables: os.environ["OPENAI_API_KEY"] or a .env file with python-dotenv. Never commit keys to GitHub — bots scan repos in minutes and will run up thousands of dollars in charges on your key.

Streaming — The "Typing" Effect

By default, the API waits until the model finishes generating, then sends the whole response. Streaming sends tokens as they're generated — creating the live "typing" effect you see in ChatGPT. It makes the UI feel much faster even if total time is similar.

Python · FastAPI + Streaming 
from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware
from fastapi.responses import StreamingResponse
from pydantic import BaseModel
import anthropic, os

app = FastAPI()
client = anthropic.Anthropic()

app.add_middleware(CORSMiddleware, allow_origins=["*"],
    allow_methods=["*"], allow_headers=["*"])

class ChatRequest(BaseModel):
    messages: list        # Full conversation history
    system: str = "You are a helpful AI assistant."

@app.post("/chat")
async def chat(req: ChatRequest):
    # Generator that yields tokens as they arrive
    def generate():
        with client.messages.stream(
            model="claude-opus-4-5",
            max_tokens=1024,
            system=req.system,
            messages=req.messages
        ) as stream:
            for text in stream.text_stream():
                yield text  # Each chunk is a few tokens

    return StreamingResponse(generate(), media_type="text/plain")

# Run: uvicorn main:app --reload --port 8000

Maintaining Conversation History

Since the AI has no memory between calls, you must send the full conversation history with every request. Your backend stores messages in a database and loads them for each new turn.

Python · Conversation Management 
# Your backend stores messages in DB, loads per user session
conversation_history = [
    {"role": "user",      "content": "My name is Sujal"},
    {"role": "assistant", "content": "Hi Sujal! How can I help?"},
    {"role": "user",      "content": "What's my name?"},
]

# The AI can answer correctly because history is included
response = client.messages.create(
    model="claude-opus-4-5",
    max_tokens=512,
    messages=conversation_history  # Always include full history!
)
# → "Your name is Sujal."

RAG — Retrieval Augmented Generation

Pinecone, Wikipedia, OpenAI Help, Mistral Docs

RAG solves the #1 limitation of LLMs: they only know what they were trained on. They don't know about your company's internal documents, last week's news, or your database. RAG lets you inject fresh, relevant context at query time — without retraining the model.

📚 FROM PINECONE'S RAG GUIDE

RAG has four core phases: (1) Ingestion — load your data into a vector database. (2) Retrieval — find the most relevant chunks for a user query. (3) Augmentation — combine retrieved data + the query into a prompt. (4) Generation — the LLM uses the context to produce an accurate, grounded answer.

Embeddings — The Heart of RAG

An embedding converts text into a list of numbers (a vector) that captures its semantic meaning. Similar sentences get similar vectors. This lets you search by meaning rather than exact keywords.

💡 Analogy (from Microsoft Azure)

Embeddings can do math: embedding("king") - embedding("man") + embedding("woman") ≈ embedding("queen"). The numerical representation captures semantic relationships — words with similar meanings cluster together in vector space.

Python · Full RAG Implementation 
from openai import OpenAI
import numpy as np

client = OpenAI()

# ① INGESTION: Convert your docs to embeddings
def embed(text: str) -> list[float]:
    response = client.embeddings.create(
        input=text,
        model="text-embedding-3-small"  # 1536-dim vectors
    )
    return response.data[0].embedding

# Your documents (in production, store in Pinecone/pgvector)
docs = [
    "Our refund policy is 30 days from purchase date.",
    "To reset your password, go to Settings → Security.",
    "We offer free shipping on orders over ₹500.",
]
doc_vectors = [embed(d) for d in docs]

# ② RETRIEVAL: Find most similar doc to user query
def cosine_similarity(a, b):
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

user_query = "Can I return something after 3 weeks?"
query_vec = embed(user_query)

scores = [cosine_similarity(query_vec, dv) for dv in doc_vectors]
best_doc = docs[np.argmax(scores)]  # Most relevant: refund policy doc

# ③ AUGMENTATION + GENERATION
response = client.chat.completions.create(
    model="gpt-4o",
    messages=[{
        "role": "user",
        "content": f"""Use this context to answer:
CONTEXT: {best_doc}
QUESTION: {user_query}
If context doesn't have the answer, say "I don't know.""""
    }]
)
print(response.choices[0].message.content)
# → "Our refund policy is 30 days, so returning after 3 weeks is within the window."

⚠️ RAG DOESN'T FULLY STOP HALLUCINATIONS

According to Wikipedia's RAG entry (citing Ars Technica): "RAG is not a direct solution to hallucination because the LLM can still hallucinate around the source material." Always instruct the model: "If the context doesn't contain the answer, say 'I don't know'." and validate critical outputs.

MCP — Model Context Protocol Nov 2024

Anthropic, Wikipedia, Google Cloud, Aerospike

MCP (Model Context Protocol) is an open standard introduced by Anthropic in November 2024. It solves the "N×M integration problem" — instead of building a custom connector for every AI model + every tool combination, MCP provides one universal protocol. Connect once, work with any AI.

📚 FROM ANTHROPIC'S ANNOUNCEMENT

MCP is described as the "USB-C port for AI applications." Just as USB-C standardises device connections, MCP standardises how AI systems connect to data sources and tools. By late 2025, the community had built over 16,000 MCP servers for services ranging from Google Drive to PostgreSQL to Slack.

The Problem MCP Solves

❌ Before MCP — N×M Problem

  • 3 AI models × 10 tools = 30 custom integrations
  • Each integration is bespoke code
  • Hard to maintain, doesn't scale
  • Adding a new tool = rewrite integrations

✅ After MCP — Universal Protocol

  • Build MCP server once, works with all AI clients
  • Claude, GPT, Gemini can all use same server
  • OpenAI and Google DeepMind both adopted it
  • 16,000+ pre-built MCP servers in community

Building a Minimal MCP Server

Python · MCP Server (install: pip install mcp) 
from mcp.server import Server
from mcp.server.stdio import stdio_server
from mcp.types import Tool, TextContent
import asyncio, httpx

app = Server("weather-server")

# ① Tell the AI what tools you have
@app.list_tools()
async def list_tools():
    return [
        Tool(
            name="get_weather",
            description="Gets current weather for any city",
            inputSchema={
                "type": "object",
                "properties": {
                    "city": {"type": "string", "description": "City name"}
                },
                "required": ["city"]
            }
        )
    ]

# ② Handle when the AI calls a tool
@app.call_tool()
async def call_tool(name: str, arguments: dict):
    if name == "get_weather":
        city = arguments["city"]
        # Call real weather API here
        async with httpx.AsyncClient() as http:
            res = await http.get(f"https://wttr.in/{city}?format=3")
        return [TextContent(type="text", text=res.text)]

# ③ Run the server (Claude Desktop connects via stdio)
if __name__ == "__main__":
    asyncio.run(stdio_server(app))

AI Agents & the ReAct Framework

IBM, PromptingGuide.ai, Salesforce, Yao et al. 2023

An AI Agent goes beyond generating text — it can take actions, observe the results, and take further actions in a loop until a goal is complete. Agents use tools (web search, code execution, databases) to interact with the real world.

💡 Analogy (from Salesforce)

Think of a detective investigating a case. A detective doesn't just guess the culprit from prior knowledge. They form a hypothesis (Thought), check the fingerprint database (Action), analyse the results (Observation), then form a new hypothesis. ReAct agents work the same iterative way — never blindly guessing when they can go and look it up.

The ReAct Framework (Yao et al. 2023)

ReAct = Reasoning + Acting. Introduced in "ReAct: Synergizing Reasoning and Acting in Language Models" (2023), ReAct interleaves Chain-of-Thought reasoning steps with tool calls. The agent alternates between Thought → Action → Observation until done.

EXAMPLE: "What's the weather in Mumbai and should I carry an umbrella?"

Thought: I need real-time weather for Mumbai. I'll use the weather tool.

Action: get_weather(city="Mumbai")

Observation: "Mumbai: 29°C, 80% humidity, Rain expected 3–6 PM"

Thought: Rain expected at 3–6 PM. I should check if the user is going out during that window.

Action: [No tool needed, I have enough info to answer]

Final Answer: "Yes, carry an umbrella. Mumbai is expecting rain at 3–6 PM today, 29°C with high humidity."

📚 FROM IBM'S REACT AGENT GUIDE

The ReAct framework uses prompt engineering to structure the agent's activity in a formal pattern of alternating Thoughts, Actions and Observations. Verbalized Chain-of-Thought reasoning helps the model decompose larger tasks into manageable subtasks. The observation phase creates a self-correcting mechanism — the agent fact-checks intermediate steps before reaching a final conclusion.

Code: Building a Simple Agent

Python · Simple ReAct Agent 
import anthropic, json

client = anthropic.Anthropic()

# Define tools the agent can use
tools = [
    {
        "name": "web_search",
        "description": "Search the web for current information",
        "input_schema": {
            "type": "object",
            "properties": {"query": {"type": "string"}},
            "required": ["query"]
        }
    },
    {
        "name": "calculator",
        "description": "Evaluate a math expression",
        "input_schema": {
            "type": "object",
            "properties": {"expression": {"type": "string"}},
            "required": ["expression"]
        }
    }
]

def run_agent(user_message: str):
    messages = [{"role": "user", "content": user_message}]

    while True:
        response = client.messages.create(
            model="claude-opus-4-5",
            max_tokens=1024,
            tools=tools,
            messages=messages
        )

        # Agent is done thinking — give final answer
        if response.stop_reason == "end_turn":
            return response.content[0].text

        # Agent wants to use a tool
        if response.stop_reason == "tool_use":
            tool_call = [b for b in response.content if b.type == "tool_use"][0]
            print(f"🔧 Using tool: {tool_call.name}({tool_call.input})")

            # Your code calls the actual tool
            result = execute_tool(tool_call.name, tool_call.input)

            # Feed result back to agent (it "observes")
            messages.append({"role": "assistant", "content": response.content})
            messages.append({
                "role": "user",
                "content": [{"type": "tool_result",
                               "tool_use_id": tool_call.id,
                               "content": result}]
            })

⚠️ AGENTS AND SAFETY

Agents that take real actions (delete files, send emails, spend money, make API calls) can cause real damage if they misunderstand. Always: (1) Add human-in-the-loop approval for irreversible actions. (2) Start with read-only permissions. (3) Log every action. (4) Set spending/rate limits on external APIs.

from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware
from fastapi.responses import StreamingResponse
from fastapi.staticfiles import StaticFiles
from pydantic import BaseModel
import anthropic, os
from dotenv import load_dotenv

load_dotenv()
app = FastAPI(title="My AI App")
client = anthropic.Anthropic()

app.add_middleware(CORSMiddleware,
    allow_origins=["*"], allow_methods=["*"], allow_headers=["*"])

class ChatReq(BaseModel):
    messages: list[dict]   # [{role, content}]
    system: str = "You are a helpful assistant."

@app.post("/chat")
async def chat(req: ChatReq):
    def stream_tokens():
        with client.messages.stream(
            model="claude-opus-4-5",
            max_tokens=2048,
            system=req.system,
            messages=req.messages
        ) as stream:
            for text in stream.text_stream():
                yield text
    return StreamingResponse(stream_tokens(), media_type="text/plain")

app.mount("/", StaticFiles(directory="static", html=True))
# Run: uvicorn main:app --reload --port 8000

let messages = []    // Full history kept client-side

async function sendMessage(userText) {
  messages.push({ role: "user", content: userText })
  renderMessage("user", userText)

  // Stream from backend
  const res = await fetch("/chat", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ messages })
  })

  const reader = res.body.getReader()
  const dec = new TextDecoder()
  let reply = ""
  const el = renderMessage("assistant", "")  // Empty bubble

  while (true) {
    const { done, value } = await reader.read()
    if (done) break
    reply += dec.decode(value)
    el.textContent = reply   // Update live as tokens arrive
  }

  messages.push({ role: "assistant", content: reply })
}
